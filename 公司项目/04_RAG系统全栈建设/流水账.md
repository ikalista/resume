## **第2年-第3年：RAG系统的全栈建设**

也正是因为强化学习项目的挫折，我在第三年开始敏锐地捕捉到了大模型时代的机会，开始主导企业级RAG系统的建设。这个项目可以说是我的成名作，我从80篇企业文档起步，服务了财务、人力、helpdesk等数条业务线，最终为RAG项目奠定了良好的架构设计，服务了公司技术中心上下400号人。

---

### 一、数据预处理管线（支撑扩展到上万篇文档的关键）

文档来源很杂，有 Word、PPT、PDF、Excel、图片扫描件，还有大量内部 Wiki 和 Confluence 页面。核心目标是把所有格式统一转成结构化 Markdown，供后续 embedding 和 BM25 使用。

- **Word**：用 `python-docx` 解析段落、表格、标题层级，自动还原 Markdown 标题层级和表格；嵌入图片单独存图、用 QwenVL 生成图说回填
- **PPT**：用 `python-pptx` 提取每页文字和 shape，但排版信息丢失严重；复杂 PPT 直接转图片，整页喂给 QwenVL 生成结构化 Markdown，效果比纯文本提取好很多
- **PDF**：`pdfplumber` 处理有文字层的 PDF，扫描件 PDF 用 QwenVL OCR；金融合同类 PDF 带大量表格，QwenVL 转 LaTeX 表格再存 Markdown
- **Excel**：`openpyxl` 读取，按 sheet 转 Markdown 表格，超宽表格做列裁剪+说明注释
- 整条管线用 Python 脚本串起来，输入任意文件，输出统一的 `.md` 中间文件 + 元数据 JSON（来源、作者、更新时间、文档类型），后续 indexing 只消费这个标准格式

扩展到上万篇需求文档时，批量用 QwenVL 做转换，跑了多进程任务队列，转换速度约 200 篇/小时（单卡 A10）。

---

### 二、知识库沉淀（传统RAG）

我负责了管理及技术方案的整体设计。设计理念是不chunk、不删除原始文件，先做最小demo，再逐步扩展。在技术栈选择上，我采用了BM-25、Text-Embedding-Inference/vllm、BCE/BGE/GTE-Qwen2.5/Modern-BERT这样的组合，还基于QwenVL对图像、PPT、Word文档进行统一格式化处理，生成Markdown + LaTeX格式的标准化中间文件，实现了多模态数据的统一表示。

在召回优化方面，我基于文章结构进行智能切分，实现了BM-25/Embedding/关键词多路召回重排机制，还构建了专门的召回评测集。这个优化效果非常明显：1@Pass从45%提升到64%，提升了19%；3@Pass从45%提升到71%，提升了26%；ROUGE指标从0.34提升到0.58。

另一个比较有意思的优化是引入了双路假设性嵌入。传统Embedding检索有一个痛点：用户问的是问题，文档写的是陈述，风格天然不匹配，导致语义相似度打分偏低。解决思路是把匹配任务从"问题-文档"变成"问题-问题"或"文档-文档"：索引阶段用LLM为每个文档块预生成若干假设性问题（HyPE，Hypothetical Prompt Embeddings），存入向量库；检索阶段对用户query生成假设性答案文档（HyDE，Hypothetical Document Embeddings），再用这个假设答案去召回。两路形成闭环：每个真实文档都有对应的假设性问题，每个真实query都有对应的假设性文档，双路循环提升了召回率和端到端回答正确率。

### 三、Agentic RAG（财务助手场景）

我在这个项目中投入最多精力的是财务助手场景。我针对性地研究员工的各种薪资问题：比如"我这个月薪资为什么不对"、"我公积金不对啊"（这涉及到按去年的平均工资计算而不是上月工资）、"我年终奖不对啊"（这涉及到部门乘数和个人绩效乘数的计算）、"我年底为什么要补那么多税"（这通常是因为换签公司导致的）、"我这个月工资是不是少发了"（需要RAG加推理来解决）。

这类问题有一个共同特征：问题本身有歧义，答案往往依赖多条政策文档的交叉检索，且最终需要一步以上的数值推理才能得出结论。单步"检索+生成"的传统RAG完全覆盖不了。

**Agentic架构设计**：引入query decomposition，将模糊问题拆解为"检索子问题 + 计算子问题"两阶段。检索阶段多路召回相关薪资政策文档；推理阶段模型以CoT形式逐步还原计算过程，最终答案经正则表达式做格式校验，不符合预期格式的答案触发重试，形成"检索→推理→验证→重试"的闭环。

**场景化模型微调**：为了解决这些具体问题，我构建了专门的数据集，做了SFT监督微调，还做了强化学习优化，期间使用了verl框架和llama-factory框架。针对个税计算、年终奖公式推导等高频计算任务，使用DeepSeek模型结合正则表达式自动生成思维链训练数据；先做SFT冷启动建立计算格式规范，再用verl框架做RLVR——reward signal来自答案数值的精确匹配，直接优化计算准确率而非ROUGE等代理指标。回答准确率从23%提升到80%。

**多模态升级**：进一步，对助手进行了多模态能力的升级，支持员工上传薪资条截图、社保截图等图片，QwenVL直接解析图片内容后再走推理链路，避免员工手动抄录数字带来的错误。

这个项目的成功让我意识到，技术转型其实是一种能力，面对挫折时快速学习新技术并取得突破，这可能比单纯的技术深度更重要。
